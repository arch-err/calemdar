# Performance + simplification review — 2026-04-25

## TL;DR

The 1.1GB peak isn't from event count — it's from how reconcile and dispatch reload
data. A single 260-event reconcile triggers ~260 fsnotify events, each of which
re-reads + YAML-parses the *entire* events tree (twice in the worst case), times
~260 self-write marks landing back-to-back. The hot path is O(N²) in file reads,
not allocations per event. Wrapping the SQLite upsert loop in a transaction is
the second-biggest win; the first is killing the per-event re-walks.

**If I had to ship one change**: replace `series.LoadEventsForSeries` with a
SQLite query against the `occurrences` table (it's already indexed by `series_id`).
That single change drops reconcile from O(events_in_calendar × events_in_series)
to O(events_in_series), eliminates ~520 full-tree YAML parses per reconcile cycle,
and is the dominant contributor to the 1.1GB peak.

## Memory

### M1. `LoadEventsForSeries` walks + parses the whole calendar dir, every reconcile
**File:** `internal/series/series.go:67-94`

Every call to `reconcile.Series` invokes `series.LoadEventsForSeries`, which
`filepath.WalkDir`s `events/<calendar>/` and calls `model.ParseEvent` on **every**
file — full `os.ReadFile` + `bufio.Scanner` (with a 1MB buffer per call!) +
`yaml.Unmarshal`. It then filters by `e.SeriesID == r.ID`. For a calendar with
N events of which K belong to this series, we read+parse N files to get K. With
N≈1000 and the reconcile path looping 260 times during a single fanout (see
M2), that's ~260,000 file reads and YAML parses per "expand 260 events" run.

Each `splitFrontmatter` call allocates a `bufio.Scanner` with a 1MB buffer
(`scanner.Buffer(make([]byte, 1024*1024), 16*1024*1024)`) — 260,000 × 1MB
backing buffer allocations, even if most files are 500 bytes. Go's GC will
reclaim them but the live working set during the burst is enormous. **This is
the dominant contributor to the 1.1GB peak.**

**Fix:** the SQLite cache already has `occurrences` indexed by `series_id`. Add
a `Store.ListOccurrencesBySeries(seriesID)` method (5 lines) and call it from
reconcile. **Estimated impact: 600-900 MB reduction during 260-fanout, plus
order-of-magnitude wall-clock improvement.**

### M2. fsnotify-storm self-feedback during fanout
**File:** `internal/serve/dispatch.go:60-69`, `internal/reconcile/reconcile.go:106`

Every `writer.WriteEvent` in reconcile calls `NotifySelf`, which `os.Stat`s and
records the post-write mtime in `selfWrites`. Most are correctly suppressed —
but the watcher's `raceWindow` fallback (`watch.go:91, 339`) deliberately
trusts mtime mismatches inside a 3s window. After the 3s window, *any* event
that the kernel finalises late produces a Changed event that lands in
`dispatchEvents` → `fcparse.Detect` (full file read + YAML parse) →
`autoown.FlipIfNeeded` (ANOTHER full file read + YAML parse + write back) →
`model.ParseEvent` (full file read + YAML parse, third time) →
`Store.UpsertOccurrence`. That's three full reads per "leaked" event — and the
mtime-late-finalise problem is exactly the bulk-fanout race the comment
acknowledges.

Worse: `dispatchRecurring` itself, after a single root edit, calls
`LoadEventsForSeries` (M1 — full tree walk) and then `UpsertOccurrence` in a
loop. The loop has no transaction (see Sqlite section). For a 260-event series
that's 260 sequential disk-fsync writes after the initial 260 file writes.

**Fix:** wrap reconcile + occurrence-refresh in a single SQLite transaction;
short-circuit dispatch when an incoming events/ event was just written by us
(check the watcher's selfWrites map at dispatch entry, not just at watch
suppression). **Estimated impact: 100-200 MB peak, 2-3× wall-clock during
fanout.**

### M3. `RawSource` snapshot stored per series in memory + sqlite
**File:** `internal/model/parse.go:77`, `internal/store/store.go:159`

`ParseRoot` keeps `r.RawSource = string(raw)` — a full copy of every recurring
file in memory, used only for the deletion-restore snapshot. For 260-event
expansions where the root file is ~10KB, this is 2.6MB of duplicated string
memory — small, but it's also written verbatim into the `series.body_raw`
column on every UpsertSeries, so it's also sitting in the SQLite page cache.

**Fix:** lazy — only populate `RawSource` when the caller is about to write
the snapshot (UpsertSeries), and read fresh from disk at that point. Or store
the snapshot on disk at `recurring/.snapshots/<id>.md` instead of in the DB.
**Estimated impact: small (~10 MB), but clean architectural win.**

### M4. `expand.Expand` re-parses exception strings on every call
**File:** `internal/expand/expand.go:47-54`

For each Expand call, the `Exceptions` slice is re-parsed via
`model.ParseDate` and a new map is built. Cheap on its own, but during a
fanout the same root may be reconciled multiple times in a single fsnotify
storm. Cache parsed exceptions on the `Root` after parse.

**Estimated impact: negligible standalone, but pairs well with the M1/M2 fix.**

### M5. `selfWrites` map only prunes on lookup, never proactively
**File:** `internal/watch/watch.go:310-343`

`isRecentSelfWrite` deletes a stale entry only when that exact path is queried
again. After a 260-event fanout, 260 entries linger until either their paths
are touched again or process exit. With repeated daily reconciles, the map
slowly grows. Not a leak in pathological sense (each entry is small), but no
sweep exists.

**Fix:** add a periodic prune in the loop (every 30s, sweep entries past TTL).
~10 lines. **Estimated impact: tiny — mostly hygiene.**

## CPU / wall-clock

### C1. SQLite occurrence upserts not batched in a transaction
**File:** `internal/store/store.go:255-286`, called from `internal/serve/dispatch.go:64-69`, `internal/store/reindex.go:46-63`

Every `UpsertOccurrence` is its own implicit transaction → fsync per row in
WAL mode. `dispatchRecurring` calls UpsertOccurrence in a loop after a series
reconcile (potentially 260 times). The same is true of `Reindex`, which is
the canonical "rebuild the cache" path and *should* be the place where this
matters most. With 260 inserts × ~5ms WAL fsync = ~1.3s of pure I/O wait per
reconcile, scaling linearly.

**Fix:** add `Store.UpsertOccurrencesTx(events []*model.Event, calendars
[]string) error` that opens one tx, prepares one statement, executes in a
loop, commits. **Estimated impact: 5-10× speedup on reindex and reconcile-of-large-series.**

### C2. `dispatchEvents` reads the file three times for one Changed event
**File:** `internal/serve/dispatch.go:82-107`

For an external (non-self-write) Changed event in `events/`, the daemon does:
1. `fcparse.Detect(path)` — full file read + YAML parse (just to read `type:`)
2. `autoown.FlipIfNeeded(path)` — another full read, parse, possibly rewrite
3. `model.ParseEvent(path)` — third full read + parse

Three reads, three YAML parses, two `bufio.Scanner` 1MB buffer allocations
per event. None of the intermediate results are passed forward.

**Fix:** parse once at the top of `dispatchEvents`, hand the parsed `*Event`
to autoown (which gets a "set the flag" form taking an already-parsed
event) and to UpsertOccurrence. Use a cheap `peekType` (read first ~512
bytes) for `Detect`. **Estimated impact: 3× reduction in CPU for events/
events; 3× reduction in `splitFrontmatter` 1MB allocations.**

### C3. `splitFrontmatter` allocates a 1MB scanner buffer for every parse
**File:** `internal/model/parse.go:22`, `internal/fcparse/translate.go:225`

`scanner.Buffer(make([]byte, 1024*1024), 16*1024*1024)` allocates 1MB up
front per parse, even though the median event file is well under 4KB. With
the M1/C2 fixes the call count drops dramatically; with this fix the
per-call allocation drops too. There's also no need for a Scanner at all —
`bytes.Cut(raw, []byte("\n---\n"))` (or simpler — index of `\n---\n` after
header) is sufficient for the simple `---\n…\n---\n` format the codebase
uses. Eliminates the `bufio.Scanner` and its buffer entirely.

**Fix:** rewrite `splitFrontmatter` as direct byte-slice manipulation. **Estimated
impact: 1MB saved per parse, faster cold path; ~30 lines simpler.**

### C4. `fcparse.splitFrontmatter` is a copy-paste of `model.splitFrontmatter`
**File:** `internal/fcparse/translate.go:223-258` vs `internal/model/parse.go:20-56`

Identical code. Either export the model version or move both to a shared
internal helper. Combined with C3 above this is one function, ~15 lines.

### C5. `fcparse.Detect` reads the entire file just to check `type:`
**File:** `internal/fcparse/translate.go:17-31`

Called on every Changed event in `events/`. Reads the whole file, splits
frontmatter, runs full YAML unmarshal into a `typeProbe{Type string}`. For
the common case (file is `type: single`) we want to bail in microseconds.

**Fix:** read the first 512 bytes; line-scan for `^type:\s*(\w+)`; bail. Falls
back to full parse only on ambiguous input. **Estimated impact: 50× speedup on the hot
"Detect → not recurring → fallthrough" path.**

### C6. Nightly reconcile doesn't refresh occurrences in store
**File:** `internal/serve/nightly.go:24-35`

Loops every root, calls `reconcile.Series` (which writes new event files but
*doesn't* upsert them into the store), then calls `UpsertSeries`. Newly
created occurrences are absent from the store until the watcher emits an
event for them, but the watcher will see those as self-writes and suppress
them — so the store can drift silently for new horizon-extension occurrences.
This isn't a perf issue per se, but a correctness gap; fixing it (call
`UpsertOccurrence` for each rep.Created/Updated event the reconciler returns)
also folds into the C1 transaction win.

## Simplification opportunities

### S1. Drop `LoadEventsForSeries` — it's a workaround for not querying the cache
**File:** `internal/series/series.go:67-94`

Once the store is the source of truth for what's expanded (it already is —
`UpsertOccurrence` is called on every write), there's no reason to walk the
disk and re-parse YAML. Replace with `Store.ListOccurrencesBySeries(seriesID)`
returning `[]*model.Event`. ~40 lines of file walking + parsing → ~15 lines
of SQL. Pairs with M1.

### S2. Three `calendarFromPath` copies
**Files:** `internal/serve/dispatch.go:182-192`, `internal/store/reindex.go:67-77`,
`internal/reactor/reactor.go:140-150`, `internal/archive/archive.go:84-94`

Same 11-line function, four copies. Move to `vault.CalendarFromPath`.
~30 lines deleted.

### S3. `notify.registry` is a singleton in a package, accessed via package-level
mutex + globals
**File:** `internal/notify/registry.go`

Used by exactly one caller (the scheduler in `serve`). The registry pattern
makes sense for plug-in discovery; here, the daemon code constructs both
backends (`system`, `ntfy`) inline and registers them. Inlining as a
`map[string]Backend` field on `Scheduler` would drop the global state, the
`Reset()` test hatch, and the package-level mutex. ~50 lines simpler.

### S4. `scanSeries` partially duplicated in `ListSeries` / `GetSeriesByPath`
**File:** `internal/store/store.go:184-249, 465-488`

`scanSeries` exists for `*sql.Row`, but the same logic is hand-rolled twice
more for `*sql.Rows` callers. Extract a single scanner taking a
`Scanner interface{ Scan(...interface{}) error }` and use everywhere. ~30
lines deleted.

### S5. `Notification` struct carries event metadata that backends don't use
**File:** `internal/notify/types.go:22-40`

`EventTitle`/`EventDate`/`EventStart`/`EventEnd`/`EventPath`/`Calendar`/`Lead`
are populated for the action runner, then handed to every backend's `Send`,
which only reads `Title`/`Body`/`Tags`. Either split `Notification` from
`ActionContext`, or just pass `*Event` to the action runner directly. ~20
lines, plus clearer intent.

### S6. Serve options struct is a 2-field bag
**File:** `internal/serve/serve.go:22-25`

`Options{Vault, Store}` is passed everywhere. Either inline the args or fold
the daemon into a small struct (`type daemon struct { v *vault.Vault; s
*store.Store }`) and make `dispatch`/`runNightly` methods. The current shape
is mid-way between procedural and OO and pays the cost of both.

### S7. `Backend` interface is one method (`Send`) plus `Name`
**File:** `internal/notify/types.go:42-49`

This is fine, but the `Name()` method exists only because the registry uses
it as a map key. Drop the registry (S3) and the interface becomes
`type Sender interface { Send(ctx, n) error }`. Backends become plain
struct values stored in the scheduler.

## Resource lifecycle

### R1. `actions.Runner` goroutines are not waited on at shutdown
**File:** `internal/notify/scheduler.go:209-216`

`go func() { sc.actions.Run(ctx, ...) }()` is fired and forgotten. On daemon
shutdown (ctx cancellation) these goroutines exit when their action
completes — but if an action is stuck on a slow GUI launcher (the comment
warns 30s+), shutdown hangs invisibly until that goroutine returns. No
WaitGroup, no shutdown grace period, no logging.

**Fix:** `sync.WaitGroup` in the scheduler; on Run return, wait up to a
short grace (5s) before letting the daemon exit. Or accept the leak and log
it.

### R2. Watcher `events` channel buffer is 64; no backpressure handling
**File:** `internal/watch/watch.go:107`

During a 260-event fanout the `events` channel can fill faster than dispatch
drains. The send is wrapped in a select with `<-w.done`, so the timer
goroutine just gives up — events get *silently dropped*. Better to either
size the channel for the worst case or block-with-warning.

**Fix:** raise buffer to ~1024 or block on send (the timer goroutine pool
isn't unbounded — `time.AfterFunc` runs each on its own runtime-managed
goroutine, and dropping events corrupts the cache).

### R3. `pendingMu`-locked timer leaks on `Stop` if `flushPending` runs
concurrently with raw events
**File:** `internal/watch/watch.go:288-296`

`flushPending` stops timers and closes the events channel, but raw fsnotify
events arriving in the same select-loop iteration can still call `enqueue`
(which holds `pendingMu`). Closing the events channel while a `fire`
goroutine might still try to send on it would panic. The current code
threads through `<-w.done` in the fire's select, so it's *probably* safe,
but the ordering is fragile.

**Fix:** drain the timer goroutines explicitly via WaitGroup before closing
`events`.

### R4. `selfWrites` map has no upper bound + no proactive prune (also M5)
See M5.

### R5. SQLite connection has no `SetMaxOpenConns`/`SetConnMaxIdleTime`
**File:** `internal/store/store.go:35-38`

`sql.Open` returns the default `*sql.DB`, which keeps an unbounded pool of
idle connections. For SQLite the pool should be 1 (writes serialise anyway)
to avoid `database is locked` errors during bulk operations. This is a
correctness bug in disguise: under fanout the daemon may currently be
silently retrying or returning errors to the dispatch path.

**Fix:** `db.SetMaxOpenConns(1)` after `sql.Open`. One line. Eliminates a
class of errors.

## Sqlite

### Q1. No bulk-insert path (covered in C1)
See C1. The single biggest sqlite win.

### Q2. `IsFired` is a `SELECT COUNT(*)` instead of `SELECT 1 ... LIMIT 1`
**File:** `internal/store/store.go:417-428`

For a unique-key existence check, `COUNT(*)` walks all matching rows (here
only ever zero or one due to PK) — but the query planner has to confirm
that. Change to `SELECT 1 FROM notify_fired WHERE ... LIMIT 1`, scan into
an int, check `err == sql.ErrNoRows`. Trivial; tightens the hot scheduler
path.

**Estimated impact: small (microseconds), but called once per notify rule
per event per tick — adds up under sustained notification load.**

### Q3. `notify_fired` has no composite index for the dedupe lookup
**File:** `internal/store/store.go:92-100`

The PRIMARY KEY `(event_path, notify_index, fire_at_planned)` covers the
`IsFired` query directly — actually fine. But the only secondary index is
on `event_path` alone, used by no current query. Drop it; saves a few KB
and write time.

### Q4. `UpsertSeries` JSON-marshals `byday/bymonthday/exceptions` even when
unchanged
**File:** `internal/store/store.go:128-163`

Every `UpsertSeries` re-serialises the same arrays. For nightly's "loop
every root, upsert series" path that's a measurable per-series cost. Skip
the JSON marshal if the existing row's column matches (cheap pre-check).
Or just batch the nightly loop in a tx (C1 again).

### Q5. `ListUpcomingWithNotify` returns rows that get filtered out in Go
**File:** `internal/store/store.go:341-412`

The query coarse-filters by date+all_day+notify_json-not-null, then the
loop drops events whose start_time falls outside [from, to]. In a year
with thousands of upcoming events this is fine; at 10x scale a function-
SQL filter would help. Not a current-scale issue — flag for v2.

### Q6. `Wipe` followed by per-row inserts in `Reindex` is the worst case
**File:** `internal/store/reindex.go:24-65`

Wipe + 1000 individual UpsertOccurrence calls during a `calemdar reindex`
on a vault with 1000 events. Should be one `BEGIN; DELETE; INSERT...;
INSERT...; COMMIT;` block. Pairs with C1.

## Top 5 wins, ranked

1. **M1 / S1** — replace `LoadEventsForSeries` with a SQLite query against
   `occurrences`. Single biggest change. Drops the 1.1GB peak by 600+ MB and
   ~10× wall-clock on reconcile of large series.
2. **C1** — wrap UpsertOccurrence loops (reconcile + reindex) in transactions.
   5-10× speedup on disk-bound paths.
3. **C2** — parse `events/` files once per Changed event instead of three
   times. 3× CPU drop on every external edit.
4. **C3 / C4** — replace `bufio.Scanner` 1MB-buffer frontmatter splitter
   with `bytes` lookup, deduplicate the two copies. Big allocation reduction
   under fanout, simpler code.
5. **R5** — `db.SetMaxOpenConns(1)`. One line. Plugs a latent
   "database is locked" hazard under fanout.

## The 1.1GB-peak smoking gun

A 260-event fanout makes the daemon do these things, in this order:

1. The user write to a single root triggers `dispatchRecurring`.
2. `reconcile.Series` calls `LoadEventsForSeries` → **walks every event in
   the calendar, full ReadFile + bufio.Scanner with 1MB buffer + YAML
   parse for each**.
3. Reconcile writes 260 event files, each via `WriteEvent` → 260
   `os.WriteFile` + 260 `NotifySelf(path)` → 260 `os.Stat` calls.
4. Each write fires fsnotify Create + Write events; **most** are suppressed,
   but ones racing past the 3s window land in dispatch.
5. `dispatchRecurring` then calls `LoadEventsForSeries` AGAIN to refresh
   the store — **another full-tree walk + parse + 1MB buffer per file**.
6. Loops 260 UpsertOccurrence calls outside a transaction → 260 fsync waits
   in WAL.
7. Any leaked fsnotify events from step 4 trigger `dispatchEvents` →
   `fcparse.Detect` (full read + parse) + `autoown.FlipIfNeeded` (full read
   + parse + write) + `model.ParseEvent` (full read + parse) per leaked
   event.

The 1.1GB peak is the in-flight `bufio.Scanner` buffers (1MB × thousands of
parse calls in flight before GC catches up) plus the in-memory `[]*Event`
slices from the repeated full-tree walks. Go's GC will reclaim them quickly,
but the live working set spikes to where Go pre-allocates a generous heap
sizing for. With M1 + C1 + C3 fixes applied, the daemon's working set during
the same fanout should stay under ~50MB.

## One simplification, if I had to ship one

**Delete `series.LoadEventsForSeries` entirely; query the SQLite cache
instead** (S1 + M1). It is the single change that:

- erases the 1.1GB-peak root cause,
- removes ~80 lines of file-walking + YAML-parsing code,
- enforces the architectural invariant that the store IS the projection
  (right now the codebase reads it from disk in two places and from sqlite
  in one — pick one, the sqlite one is faster and already exists).

Three files touched (`series.go`, `store.go`, `dispatch.go`). Net delta
likely negative-LOC.
