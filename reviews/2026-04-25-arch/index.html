    Architecture Review — calemdar v1¶

    Date: 2026-04-25 Reviewer: arch-err (architecture pass) Scope: read-only structural review of internal/ + cmd/calemdar/ after the v1 ship. Code-quality nits and perf are out of scope (covered by the parallel reviewers).

    State of the architecture (5 lines)¶

    The package layout mostly mirrors the data flow, and the source-of-truth invariant ("markdown is truth, sqlite is a projection") is honoured almost everywhere — the auto-restore path is the deliberate, well-marked exception. internal/serve is doing more than orchestration: it owns dispatch routing, notify backend wiring, the nightly loop, and per-event reconcile fan-out, so it has become the de-facto coupling point. Concurrency is small (4 goroutines) and shutdown is mostly clean, but a few callbacks (watcher debounce timers, scheduler action goroutines) escape the ctx tree. The biggest structural risks are global state in model + config.Active, sync-safety assumptions that hold for one writer but not for syncthing, and silent-failure paths in the dispatch loop where a reconcile crash is logged but never escalated.

    What I'd refactor first if I had a week¶

    I'd split internal/serve into a daemon package whose only job is to wire goroutines and propagate ctx, plus a new dispatcher package that owns the event-loop switch. Then I'd promote a Plan type out of reconcile so the "compute plan" / "apply plan" steps become inspectable, partial-failure-safe operations. While in there, I'd kill the package-level model.Calendars, model.defaultLocation, and config.Active mutables — pass a *model.Env (or similar) explicitly to every entry point so tests and multi-vault scenarios stop fighting global state. None of this changes behaviour; it just means a partial reconcile failure no longer leaves the store and the disk in disagreement, and a future "calendar B" command can run without nuking the daemon's view of calendar A.

    Structural concerns¶

    1. internal/serve is overloaded¶

    • Where: internal/serve/serve.go, internal/serve/dispatch.go, internal/serve/nightly.go.
    • Current state: the serve package owns (a) goroutine wiring (watcher → dispatch loop, nightly timer, scheduler), (b) the dispatch switch routing fsnotify events to reconcile/reactor/store/backup, (c) backend registration for the notify subsystem (it imports notify and knows how to instantiate System and Ntfy), (d) the nightly run-everything routine, (e) helper math (parseHHMM, nextNightly, calendarFromPath). It's the only internal/ package that imports more than five siblings (currently 11 in dispatch.go + serve.go combined).
    • Recommendation: split into three packages.
    • daemon (small): Run, ctx wiring, goroutine lifecycle.
    • dispatch (medium): the Event → action router. Tests already exist here (dispatch_test.go) — moving them out of serve clarifies the seam. Becomes the natural home for calendarFromPath, which today is duplicated in serve/dispatch.go, archive/archive.go, and store/reindex.go — three near-identical copies.
    • notifyboot (small) or move to a notify.WireFromConfig(...) helper: the startScheduler function is purely "config → registered backends", which doesn't belong in the daemon orchestrator.
    • Risk addressed: every new feature today needs a serve edit, which raises merge-conflict surface and makes it harder to test the daemon in isolation. As more backends/sources land, the godoc-ability of serve collapses.

    2. calendarFromPath is duplicated three times¶

    • Where: internal/serve/dispatch.go:182, internal/archive/archive.go:84, internal/store/reindex.go:67, internal/reactor/reactor.go:140. Four copies, technically.
    • Current state: all four have the same shape (rel from EventsDir, split on /, return first part), with slight error-handling drift — reactor returns ("", err) on bad relpath, the others swallow to "?".
    • Recommendation: promote to vault.CalendarFromEventPath(path) since it's a pure path operation tied to vault layout.
    • Risk addressed: if the vault layout ever changes (e.g. nesting events under year/), four call sites need synchronised edits, with a real chance one drifts and silently writes events under calendar "?".

    3. Global mutable state in model and config¶

    • Where: internal/model/calendar.go (Calendars, SetCalendars), internal/model/date.go (defaultLocation, SetTimezone), internal/config/config.go (var Active = Defaults()).
    • Current state: every package that needs the timezone or calendar list reads the package-global model.Location() / model.ValidCalendar(...), and the daemon mutates these at startup via main.loadConfig. Tests reset config.Active = config.Defaults() (see dispatch_test.go:27, reconcile_test.go:26, cmd_recurring_test.go:37) — proof that the global is leaky enough to need explicit zeroing.
    • Recommendation: introduce an Env (or Runtime) struct carrying Location *time.Location, Calendars []string, and the resolved config; thread it through the entry points (reconcile.Series, expand.Expand, archive.Run, serve.Run, etc.). Keep the globals as a thin facade for one release if needed.
    • Risk addressed: today, anything that reconfigures timezone mid-process can race with an in-flight reconcile (no mutex around defaultLocation). More important, it blocks future capability — a "preview against tz X" CLI flag can't be implemented without forking the global.

    4. notify.registry is a package-level singleton¶

    • Where: internal/notify/registry.go.
    • Current state: registered backends live in a package-global map; the daemon's startup calls notify.Reset() then notify.Register(...). This works for the single-daemon design but means every test that touches the scheduler must order: reset → register → run, which the tests do manually.
    • Recommendation: make registry an instance carried on Scheduler (e.g. (*Scheduler).RegisterBackend). The package-level helpers stay for convenience but are a thin wrapper.
    • Risk addressed: singletons make multi-vault, multi-test, and future in-process API scenarios harder. Also lets us drop the comment "Reset is test-only" — instances are inherently per-test.

    Concurrency¶

    5. Action subprocess goroutines escape ctx tracking¶

    • Where: internal/notify/scheduler.go:209-216.
    • Current state: when a notify rule has an action:, the scheduler spawns a go func() { sc.actions.Run(ctx, ...) }() and never tracks it. On daemon shutdown (ctx.Done()), Run returns from the tick loop, but any in-flight action has its ctx cancelled mid-spawn — the goroutine itself is not waited on. With MaxConcurrentSpawns = 4 and an action timeout of up to 30s, a SIGTERM during a fire window can leak up to 4 actively-running children that the parent never reaps. Worse: defer s.Close() in runServe closes the SQLite handle while those goroutines may still call into it via the runner's logging path (currently they don't, but it's one log-line away).
    • Recommendation: carry a sync.WaitGroup on the scheduler; track every async action; Run waits the WG before returning. Add a shutdown grace window (e.g. 10s) before forcefully cancelling.
    • Risk addressed: torn child processes, half-written log lines, and future use-after-close on the store handle.

    6. Watcher debounce timers are not ctx-bound¶

    • Where: internal/watch/watch.go:247,251 (time.AfterFunc), flushPending at 288.
    • Current state: Stop() closes done, the loop returns, and flushPending calls timer.Stop() on every pending timer — but time.AfterFunc returns false from Stop() if the func has already started; since each callback grabs pendingMu and tries to push onto events, you can race with the channel being closed in flushPending. The select-on-done in fire() (line 282) protects the channel-write side, so we won't write after close, but the debounce callback can still walk through the lock while the channel is mid-close. No data corruption observed, but it's a fragile shutdown.
    • Recommendation: track in-flight fire goroutines with a WG (the time.AfterFunc returns a *Timer, but the callback runs in its own goroutine), or move the fire logic onto the loop goroutine and use the timer purely for scheduling (e.g. a time.Timer reset onto a channel the loop selects on).
    • Risk addressed: pre-shutdown event delivery, deterministic test behaviour for the watcher.

    7. selfWrites map is unbounded under bulk reconcile¶

    • Where: internal/watch/watch.go:64,153,318.
    • Current state: every self-write call inserts into the map; entries are only deleted on a lookup that finds an expired entry. If the reconcile path writes 260 files (the comment near raceWindow notes this is a real number), and a few of those events never reach the watcher (Obsidian sync touched mtime independently, kernel coalesced), those entries stay in the map until something else touches that exact path. Over weeks of nightly reindexes this leaks slowly.
    • Recommendation: TTL sweep — janitor goroutine running every minute that walks the map and drops entries older than 2×TTL. Trivial addition; the lock is already there.
    • Risk addressed: memory growth in long-lived daemon, hard-to-debug "false suppression" of much later edits to a stale-marked path.

    8. Scheduler.lastTick mutex is correct but the structure is weird¶

    • Where: internal/notify/scheduler.go:115-117,165-167.
    • Current state: tick is only ever invoked from the single goroutine that runs Scheduler.Run. The mutex around lastTick is defensive against a future external trigger, not actually needed today.
    • Recommendation: either keep the mutex and document that the scheduler will be externally triggerable (e.g. for tests), or drop it. Right now it implies sharing that doesn't exist. Low priority.
    • Risk addressed: misleading reader.

    Sync resilience¶

    The vault is syncthing-replicated. v1's design assumes one writer per file at a time; that holds in practice but the code doesn't defend it.

    9. No atomic-rename writes¶

    • Where: internal/writer/writer.go:79-103.
    • Current state: writeMarkdown calls os.WriteFile, which is a truncate-then-write. If the daemon crashes, the kernel power-fails, or syncthing replaces the file at the exact wrong moment, an event file can land empty or partial. Obsidian then refuses to parse it; the reconcile next time will create a new occurrence beside it.
    • Recommendation: write to <path>.tmp, fsync, then os.Rename. The watcher already ignores non-.md files (line 219), so the temp doesn't trigger spurious events. Rename is the standard atomic primitive — also fixes a bunch of syncthing-induced "saw a half-file" quirks.
    • Risk addressed: torn writes in the markdown that the user is trying to edit, and "phantom" duplicates appearing during outages.

    10. series_id collision is impossible-by-design but never enforced¶

    • Where: internal/model/parse.go:59 (no validation of id), internal/store/store.go:51 (series.id PK).
    • Current state: if two devices independently create a recurring root with the same id (a copy-paste from a template, or the user edits a UUID by hand), syncthing will deliver both. Whichever one reconciles first wins; the second UpsertSeries overwrites the row silently. The two root_path values disagree though — that column has UNIQUE, so you'll get a constraint error on the second insert (good for visibility, but the user-facing behaviour is "calemdar randomly errors on a healthy-looking file").
    • Recommendation: at parse time, validate id is a UUID. At reconcile time, if GetSeries(id) already returns a different root_path, surface a structured "conflict" log and skip the reconcile. The auto-restore path already proves we have the vocabulary for non-fatal disk/store divergence.
    • Risk addressed: sync-induced confusion when two roots claim the same id, especially when the conflicting file came in via syncthing rather than the local user.

    11. Self-write suppression is mtime-based; cross-device clock skew breaks it¶

    • Where: internal/watch/watch.go:144-167,310-343.
    • Current state: NotifySelfWrite stats the file post-write, records mtime, then later events compare incoming mtime to the recorded mtime to decide "ours or theirs". Within one host this works. But if a syncthing peer rewrites the file with its mtime preserved (the default), our suppression logic sees mtime == ours and silently skips — meaning an external edit from a peer can be missed. The comment block discusses kernel mtime races but doesn't consider remote mtime forgery.
    • Recommendation: use file content hash (sha256 of body) instead of mtime as the post-write fingerprint. We're already reading the file body to write it; hashing on write costs nothing. Compare hash on fire — same cost as a stat. Fully immune to mtime games.
    • Risk addressed: silently dropped peer edits during sync windows. This is one of the most likely v1 bug sources because it only shows up when two devices edit the same root in close succession.

    12. notify_fired dedupe is per-device¶

    • Where: internal/store/store.go:417-441.
    • Current state: the dedupe table lives in <vault>/.calemdar/cache.db, which is documented as "intentionally NOT synced across devices" (good — sqlite WAL would corrupt under syncthing). But that means if the user runs the daemon on two laptops, both will fire each notify rule independently. For a single user with one daemon at a time this is fine; for laptop+desktop setups it's spam.
    • Recommendation: v1 lockfile pattern — <vault>/.calemdar/lock with the daemon's hostname and PID; on startup, refuse to schedule if another host's lock is fresh (< 2× tick). User-visible message: "another daemon is running on host X; only one daemon can deliver notifs". Sync-friendly because the lockfile is in .calemdar/ (not synced) — wait, that means we need a synced coordination point. Move the lock to <vault>/recurring/.calemdar-notify-lock or similar. Honest answer: this is a v2 problem, but worth a one-paragraph note in the README so users don't get surprised.
    • Risk addressed: double notifications when the user (very common) has the daemon set to autostart on every machine.

    13. body_raw snapshot trusts whichever device last upserted¶

    • Where: internal/store/store.go:128-163, internal/serve/dispatch.go:143-179 (auto-restore).
    • Current state: the snapshot is rewritten every UpsertSeries. If device A has a stale view (missed a sync round), it can overwrite the snapshot with old contents. Subsequent external delete on device B triggers a restore to A's stale version on whichever device the delete fires on. The RawSource field carries no timestamp.
    • Recommendation: record body_raw_mtime alongside body_raw, and on restore prefer the larger of (sqlite snapshot mtime, latest backup-file mtime). The backup file already has a timestamp in its name; this is just leveraging it.
    • Risk addressed: the auto-restore stomping a peer's recent edit with our own outdated copy — silent data loss exactly when the user was hoping the safeguard would protect them.

    Extensibility¶

    The design promises "new notif backend = 1 file + 2 config fields". Let's test that against the code.

    14. Adding a notif backend is closer to 4 touch points than 1¶

    • Where: to add backend slack, today you need to:
    • Write internal/notify/slack.go (the backend itself). ✓ "1 file"
    • Add SlackBackend struct to internal/config/config.go's Backends. ✓
    • Wire merge logic for it in mergeBackends. ✗ "2 config fields" becomes ~8 lines of merge boilerplate.
    • Wire registration in internal/serve/serve.go:startScheduler — another if cfg.Backends.Slack.Enabled { ... Register ... }. ✗
    • Plumb it through cmd/calemdar/cmd_notify.go (notify test). ✗
    • Current state: the "1 file" promise is the backend implementation only; the wiring fans out. internal/serve/startScheduler knows every backend by name. Same for the merge function.
    • Recommendation:
    • Make config.Backends a map[string]json.RawMessage (or map[string]any) keyed by backend name. Each backend exposes a Decode(raw) (Backend, error) constructor + a name. Registration is then a loop in startScheduler: for each enabled backend in the config map, look up the constructor in a backend-factory registry, decode, register.
    • This collapses 5 touch points to 2: the backend file + a notify.RegisterFactory("slack", slack.New) line in an init().
    • Risk addressed: the cost-to-add-a-backend is what determines whether new backends actually get added. As-is, slack/email/discord will each be a 30-min job; with the factory pattern it's 5 minutes.

    15. Adding a freq type is 4-5 touch points¶

    • Where: internal/model/freq.go, internal/expand/expand.go, internal/expand/dates.go, internal/fcparse/translate.go, internal/fcparse/rrule.go.
    • Current state: freq is a string constant compared with == in the expand package's switch. Adding yearly means extending all five files, which is fine, but more interesting: the only check is model.ValidFreq and the switch in Expand. There's no test that asserts "every valid freq is handled by Expand" — adding a constant to freq.go and forgetting expand.go panics with no error, only a nil events slice (the switch falls through).
    • Recommendation: restructure Expand so its switch returns error on unknown freq (it currently doesn't — it silently returns empty), and add a tabular test that loops every freq in ValidFreq and runs a minimal expansion to flush the "did you forget to wire this up" check.
    • Risk addressed: silent "no occurrences" bugs when a freq is added but expansion isn't.

    16. New calendar source format isn't extensible¶

    • Where: internal/fcparse/.
    • Current state: fcparse.Detect knows about recurring and rrule. Adding ICS import (a likely future ask) requires a new package and a new dispatch case in dispatchEvents (serve/dispatch.go:90). No interface separates "kind detection" from "translation".
    • Recommendation: a sourceformat package with a registry (format.Register(name, Detector, Translator)) that the dispatch layer iterates. Cheap to add but unlocks "ICS feed import", "iCal subscription", etc.
    • Risk addressed: structural debt before the second source format shows up.

    Failure modes¶

    17. Dispatch loop swallows reconcile errors and continues¶

    • Where: internal/serve/serve.go:82-85, internal/serve/dispatch.go:49-69.
    • Current state: if reconcile.Series returns mid-fanout (say, a writer.WriteEvent failed because the disk briefly went read-only), the partial reconcile state is left on disk, the store has some of the new occurrences upserted but not the rest, and the daemon logs "dispatch foo: write /path: " then loops back to the next event. There is no structured "I'm in a bad state" signal — the user only learns by manually inspecting logs. There is no retry/backoff.
    • Recommendation:
    • Bubble structured errors from reconcile (a Report already exists — add a Failures []FailedWrite field).
    • On any reconcile failure, mark the series as "needs-rerun" in the store (a series.dirty INTEGER) and retry on the next nightly OR on the next event referring to that series.
    • Surface a "calemdar status" / "calemdar health" CLI command that reads the dirty flags and lists trouble spots.
    • Risk addressed: silent partial-state where the user thinks the daemon is healthy because nothing crashed, but their calendar is half-correct.

    18. SQLite corruption has no recovery path¶

    • Where: internal/store/store.go:30-45, cmd/calemdar/cmd_serve.go:18-22.
    • Current state: if cache.db is corrupt, store.Open returns the error, runServe returns, the daemon exits. systemd restarts it; it fails again; loops forever. The user has to know to rm cache.db && calemdar reindex.
    • Recommendation: on Open error, attempt a cache.db.bad move + Reindex from scratch. Log loudly. If the second attempt also fails, exit. The reindex path is already idempotent — it's the easiest recovery move possible.
    • Risk addressed: boot loop on a single-bit-flipped sqlite file.

    19. Watcher crash silently kills the daemon's reactivity¶

    • Where: internal/watch/watch.go:188-206 (loop returns on channel close), internal/serve/serve.go:78-81 (dispatch loop also returns on closed event channel).
    • Current state: if fsnotify fails (inotify-watch limit hit, NFS edge case, kernel quirk), the watcher's fs.Events channel gets closed; loop returns; events channel gets closed in flushPending; the dispatch loop in serve.go:78-81 sees !ok and returns nil from Run. The daemon exits cleanly — systemd doesn't see an error to log. The user observes "calemdar serve doesn't react to my edits anymore", restarts the service, and it might or might not come back.
    • Recommendation: in serve.Run, treat watcher channel close while ctx is still alive as an error (return errors.New("watcher closed unexpectedly")). Better, attempt to restart the watcher up to N times before giving up — the inotify-limit case in particular is often transient.
    • Risk addressed: "the daemon is up but doing nothing", the worst failure mode for a sync tool.

    20. PruneOlderThan config is dead¶

    • Where: internal/notify/scheduler.go:32-34, internal/serve/nightly.go:48-54.
    • Current state: SchedulerConfig.PruneOlderThan is documented and defaulted, but the actual prune is called from nightly.go with a hardcoded 14 * 24 * time.Hour cutoff. The Scheduler-level config field is never read.
    • Recommendation: either remove the field from SchedulerConfig (it's misleading) or wire nightly.go to read it through config.Active.Notifications (and add a config field for it). If removed, document that prune cadence is daemon-controlled, not scheduler-controlled.
    • Risk addressed: misleading config surface; user adjusts a knob that has no effect.

    21. notify_fired records firing-attempt, not delivery success¶

    • Where: internal/notify/scheduler.go:173-193.
    • Current state: RecordFired is called before the per-backend Send. The comment ("Record first — better to suppress a future replay than to double-fire after a crash") is honest, and a defensible call. But it means: if notify-send is broken on this machine, the scheduler logs "send failed" once, records the fire, and never retries. A user who fixes notify-send an hour later doesn't get the missed notification — the dedupe table has it marked as fired.
    • Recommendation: add an outcome TEXT column ('fired_ok' / 'send_failed' / 'no_runner') so a future sweeper or CLI can re-fire the failures. Or accept the tradeoff and document it prominently — there's a real argument for "no retries in v1, full stop".
    • Risk addressed: silent missed notifications when a backend has a recoverable transient failure.

    22. handleFCRecurring runs the reactor on the whole tree per event¶

    • Where: internal/serve/dispatch.go:111-131.
    • Current state: any single FC-style file save in events/ triggers a full reactor.Run(opts.Vault) walking every md file in events. Comment says "Cheap enough for now and always idempotent". At v1 vault sizes (a few hundred files) this is fine. At ten thousand files this is O(N) per save, which is a problem because it's also re-detecting types via fcparse.Detect, which re-reads every file.
    • Recommendation: limit the walk to the directory of the changed file (it's always a single calendar). The 99% case is a one-file migration.
    • Risk addressed: future-vault-size DOS by self.

    23. dispatch doesn't acquire a per-series lock¶

    • Where: internal/serve/dispatch.go.
    • Current state: the dispatch loop is single-threaded (for { select { case ev := <-w.Events() ... } }), so two concurrent dispatches can't run. Good. But series.LoadEventsForSeries walks the disk while a parallel write goes through WriteEvent (also during dispatch) — fine because it's the same goroutine. If the scheduler ever shares state with dispatch (currently doesn't, but the scheduler reads from the store, which dispatch writes to), we'll need cross-cutting locking.
    • Recommendation: keep the single-goroutine dispatch invariant documented at the top of serve.Run; if the scheduler ever needs to write to the store on its own (today it only reads + writes notify_fired), reconsider. Low-priority structural note.
    • Risk addressed: future regression from making dispatch concurrent without realising downstream code assumes single-writer.
