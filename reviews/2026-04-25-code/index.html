    Code Review — 2026-04-25¶

    Calemdar is in solid shape for v1: layering is clean, the internal/ boundaries are consistent, and the recent fixes (autoown race, scheduler 0-lead window, recurring delete cascade) are well-tested. The codebase is mostly idiomatic Go with sensible error wrapping and few gotchas. The biggest hardening opportunities are around the watcher/scheduler boundary (TTL pruning, race window assumptions, panic-safe goroutines), a handful of latent autoown footguns, and a couple of SQL/map-iteration bugs that are pure one-line fixes. No security-grade defects; the threat model around actions is honoured.

    Total findings: 47 — 6 Critical, 19 Important, 22 Nice-to-have.

    Critical¶

    C1. Scheduler tick advances lastTick even on store-query failure¶

    internal/notify/scheduler.go:129-167 — when ListUpcomingWithNotify errors, tick returns early without updating lastTick (good), but every other failure path inside the loop continues to fall through to sc.lastTick = now. A transient SQLite error on the upcoming query is fine, but a malformed Notify JSON that escapes the per-row continue (e.g. a future scan error mid-loop) would still update lastTick to now and lose the missed window. Fix: defer lastTick = now only when the whole tick reaches end-of-loop cleanly; on any aborted iteration, leave lastTick alone so the next tick re-scans.

    C2. dispatchEvents flips user-owned for events the scheduler is mid-firing¶

    internal/serve/dispatch.go:94-106 — autoown.FlipIfNeeded is called on every Changed event in events/, including events that the scheduler itself just touched via the action runner (e.g. an action that opens the event in nvim, mutating mtime). Today's autoown false-flip race fix (watch.raceWindow) protects against the scheduler/writer path but not action-side mutations. Fix: skip FlipIfNeeded when e.SeriesExpandedAt is within the last raceWindow of time.Now() and the file's mtime difference is below the same threshold; or gate the flip on a stat-then-stat compare bracketing the parse.

    C3. selfWrites map grows unbounded — no janitor¶

    internal/watch/watch.go:62-67, 137-166, 310-343 — entries are written by NotifySelfWrite / NotifySelfDelete and only removed when an isRecentSelfWrite(path) lookup notices the TTL has elapsed. Paths that are written once and never touched again (e.g. archived events) sit in the map for the lifetime of the daemon. After a long-running session with bulk reconciles, this can hold tens of thousands of stale entries. Fix: add a periodic janitor goroutine that walks selfWrites every minute and deletes entries older than ttl; or piggyback on the nightly loop.

    C4. fire's action goroutine captures ctx, which is the per-tick context¶

    internal/notify/scheduler.go:209-216 — the go func() { sc.actions.Run(ctx, ...) }() captures the ctx passed into tick. That ctx is the daemon ctx today (passed through Run → tick), so this is correct in practice — but it's brittle. If a future refactor introduces a per-tick context.WithTimeout, the action goroutine survives the tick return while its ctx gets cancelled. Fix: pass the daemon's root ctx into Scheduler once at construction time and reference sc.rootCtx inside the goroutine, not the per-tick ctx.

    C5. runRecurringDelete opens the store after the file delete¶

    cmd/calemdar/cmd_recurring.go:166-185 — if os.Remove(r.Path) succeeds and store.Open(v) then fails (lock contention, fs full, journal corruption), the on-disk root is gone and the sqlite series row + occurrence rows are stale; a subsequent serve reindex would rebuild correctly only if the user reads the warning and runs reindex. The series row is also the only body_raw snapshot — the very recovery path the daemon uses to auto-restore. Fix: open the store before removing the root file; perform sqlite cleanup only after a successful file delete; on store-open failure abort the delete with a clear error before any disk mutation.

    C6. ListUpcomingWithNotify widens the SQL date filter by ±1 day but doesn't widen for tz that crosses midnight when from > to is impossible¶

    internal/store/store.go:341-412 — the comment claims a ±1 day widen covers tz straddling, but to.AddDate(0, 0, 1) is correct only for forward-tz. For UTC+12 events stored as 2026-05-04 with start_time 23:30, a query window of [2026-05-04 22:00 UTC, 2026-05-04 23:00 UTC] widens to 2026-05-03..2026-05-05 — fine. But events with Date produced by e.Date+" "+e.StartTime parsed in loc produce a wall-clock time that gets compared against UTC from/to from the scheduler. Scheduler passes from = last, to = now.Add(maxLead) — both are wall-clock time.Time from time.Now() (UTC-internal but locale-tagged via timer). The fine-filter line 406 (ts.Before(from) || ts.After(to)) compares two wall-clock times in possibly different Location()s; time.Time.Before is monotonic and tz-agnostic so this is probably correct, but the ±1 day SQL widen is anchored to from and to formatted as 2006-01-02 — which uses their location, not model.Location(). Fix: format the SQL widen using from.In(model.Location()).AddDate(0,0,-1).Format(...) so the date string matches the timezone the events were authored in.

    Important¶

    I1. parseEventStart silently drops events whose date+time can't parse¶

    internal/notify/scheduler.go:279-288 — returns (zero, false), scheduler continues with no log. A valid-looking row with garbage start_time is a configuration bug worth surfacing. Fix: log at warn level on parse failure with row.Event.Path so the user sees the issue.

    I2. recordFired failure does NOT block Send — double-fire risk on restart¶

    internal/notify/scheduler.go:174-179 — comment says this is intentional, but the failure mode silently drops a sentinel: if SQLite is locked momentarily, the notification fires but no row lands. The next tick replays the rule. Fix: keep current behaviour but bump the log to a clear WARN: dedupe sentinel not recorded — possible duplicate on restart so the user can correlate noisy push duplicates to a real cause.

    I3. RecordFired writes RFC3339 (second resolution) but IsFired queries the same key — sub-second skew loses dedupe¶

    internal/store/store.go:417-441 — fireAt.UTC().Format(time.RFC3339) truncates to seconds. If fireAt is computed twice with different sub-second values (it shouldn't be — startTs.Add(-lead) is deterministic per row+rule+tick), the keys still match. But if MinLead is reduced or sub-minute leads ever land, the dedup key may collide across rules. Fix: include the rule lead string in the key, e.g. add a lead column to notify_fired or fold into fire_at_planned as lead@time.

    I4. handleRecurringDelete writes the file with 0o644 even when source had different mode¶

    internal/serve/dispatch.go:170 — os.WriteFile(path, []byte(r.RawSource), 0o644) always sets 0o644. Same in CLI restore (cmd_recurring.go:233). For Obsidian on Linux that's fine, but if the user has a stricter umask or the original mode was 0o600 (secrets, etc.), it leaks. Fix: stat the original mode before delete, store it in the snapshot, and apply on restore. Or document the policy and stick with 0o644.

    I5. addTree swallows errors from fs.Add for newly-created dirs¶

    internal/watch/watch.go:208-216 — if info, err := os.Stat(...); err == nil && info.IsDir() { _ = w.addTree(ev.Name) }. The _ = discards addTree errors; if a newly-created dir can't be watched, events under it vanish silently. Fix: log on failure: if err := w.addTree(ev.Name); err != nil { log.Printf("watch: addTree %s: %v", ev.Name, err) }.

    I6. flushPending closes w.events while fire may still be writing to it¶

    internal/watch/watch.go:255-296 — flushPending is called from the loop() goroutine on <-w.done, but fire (running on time.AfterFunc goroutines) holds pendingMu to delete its entry then releases the lock and writes to w.events. If Stop() races the timer's fire, fire writes to a closed channel → panic. Fix: in fire, check select { case <-w.done: return; default: } after acquiring pending entry, before sending.

    I7. enqueue clobbers the original source classification on re-trigger¶

    internal/watch/watch.go:243-253 — on re-fire, p.kind = kind; p.src = src. Source can never change for a path (recurring vs events is path-prefix), so this is harmless today. But the silent overwrite is a footgun if the classification ever broadens. Fix: assert source equality and log if mismatch; or skip the assignment — the original is fine.

    I8. dispatchRecurring re-loads events from disk after every reconcile¶

    internal/serve/dispatch.go:60-69 — calls series.LoadEventsForSeries(opts.Vault, r), which does a full WalkDir over events/<calendar>. For a calendar with thousands of events (life), that's an O(N) walk on every single reconcile event. The reconcile pass already produced the in-plan list. Fix: have reconcile.Series return the events it wrote so dispatch can UpsertOccurrence from that list directly.

    I9. archive.Run calls writer.NotifySelf(target) for archive-tree paths that aren't watched¶

    internal/archive/archive.go:73-74 — comment says "destination under archive/ (not watched, but harmless)", but the watcher's selfWrites map grows for every archived event with a path that will never be re-touched (compounds C3). Fix: drop the second NotifySelf call.

    I10. reactor.migrateOne calls NotifySelf on the original path AFTER the file is removed¶

    internal/reactor/reactor.go:120-123 — os.Remove(path) runs, then writer.NotifySelf(path). NotifySelf is the post-write hook (stats the file); the file is gone so it falls into the mark.isRemove = true branch. Functionally OK, but semantically should be NotifySelfDelete(path) before the Remove. Fix: swap to NotifySelfDelete and move it above the os.Remove.

    I11. Names() in actions.Runner has unspecified ordering¶

    internal/actions/actions.go:182-193 — comment says "Order is unspecified"; CLI notifyActions iterates cfg.Actions directly, so users see different orderings between runs. Cosmetic but jarring. Fix: sort.Strings before return.

    I12. Reload swaps cfg but doesn't resize the semaphore¶

    internal/actions/actions.go:163-167 — comment says "keeps the runner's semaphore intact". If the user changes max_concurrent_spawns, they need a daemon restart. Document this in the function comment and surface a log line, or rebuild the semaphore atomically. Fix: log a warning if a Reload would change concurrency and the runner can't honor it without restart.

    I13. Reset() is documented as test-only but is called in serve.startScheduler¶

    internal/notify/registry.go:30-35 — comment: "Test-only helper. Production code never needs to drop registered backends." But serve.go:138 does exactly that ("Always start fresh"). Either the production call is wrong, or the comment is stale. Fix: update the comment to reflect that Reset is also called on daemon (re)start.

    I14. runRecurringDelete doesn't check for in-progress watcher¶

    The CLI delete races against a running daemon. The watcher's debounce (500ms) means a CLI delete during a tick window can still produce a Deleted event in dispatch — which would then look up by path, find no series row (we just deleted it), and log "no snapshot". Functionally safe but the warning is misleading. Fix: in handleRecurringDelete, check whether a backup file exists for this path's slug created in the last ~5s before logging "no snapshot" — if so, this is the CLI delete, log a debug-level message instead.

    I15. splitFrontmatter (parse.go and fcparse) is duplicated verbatim¶

    internal/model/parse.go:20-56 and internal/fcparse/translate.go:223-258 — two copies of the same scanner. They have identical bugs/quirks (16MB max). Fix: move to a small shared package (e.g. internal/markdown) and import. The "fcparse doesn't depend on model's unexported helper" reasoning isn't load-bearing.

    I16. daysBetween uses int(b.Sub(a).Hours() / 24) — DST disagreement¶

    internal/expand/dates.go:140-142 — Hours()/24 floors negative durations toward zero, which is not the same as toward minus-infinity. For the weekly expander, daysBetween(anchor, cur) where cur < anchor produces a wrong week count and a wrong modulo result. The current call sites have cur >= anchor so it works, but it's a footgun if weekly is ever called with a window before startDate. Fix: use int(b.Sub(a) / (24 * time.Hour)) — same floor — but document the assumption or guard against negative.

    I17. monthly modulo on (aligned - anchorYM) % interval may align below window¶

    internal/expand/dates.go:63-96 — when winYM == anchorYM, aligned == anchorYM (mod 0), correct. When winYM > anchorYM and the diff isn't divisible, aligned += interval - mod — bumps past winYM. Looks right. But if winYM < anchorYM { winYM = anchorYM } mutates winYM after aligned is initialized — actually aligned is set on the next line from winYM, so OK. Confusing read; refactor into a helper for readability.

    I18. isRecentSelfWrite returns true on stat error → masks unrelated deletes¶

    internal/watch/watch.go:323-329 — info, err := os.Stat(path); if err != nil { return true }. This is intentional: a self-write whose post-write stat fails (file already removed) shouldn't be re-emitted. But if a separate, unrelated path lookup hits a stat error (permission denied, transient FS hiccup), we suppress real events. Fix: distinguish ErrNotExist (genuine race with removal — return true) from other errors (log and return false).

    I19. fire calls os.Stat after debounce — does not preserve Deleted for paths that just-now exist¶

    internal/watch/watch.go:273-281 — when a Deleted event is queued and then the file gets re-created within the debounce window, fire will see the file via os.Stat and emit Changed. That's correct for atomic-rename saves but wrong for "user deleted, then re-created with different content" — the Deleted is lost. Niche, but it can mask the recurring-delete safeguard if the user does a quick delete+recreate via Obsidian. Fix: if p.kind == Deleted and a file now exists at path, emit both a Deleted then a Changed.

    Nice-to-have¶

    N1. parseEventStart builds a string with Sprintf per-row¶

    internal/notify/scheduler.go:283 — concat e.Date + " " + e.StartTime is faster and simpler. Fix: replace fmt.Sprintf("%s %s", ...) with a + concat.

    N2. formatBody doesn't pluralise "now" / handle "1m"¶

    internal/notify/scheduler.go:257-275 — output reads "Standup — in 1m @ 10:00" which is fine, but "in 0m" is impossible (we map to "now"). Edge case: "in 60m" should arguably display as "in 1h". Fix: if duration parses cleanly, use lead.String() after the parse; or keep raw string for fidelity and document the choice.

    N3. buildNotification always sets Title to "calemdar: " + row.Event.Title — no override¶

    internal/notify/scheduler.go:226-238 — phones truncate at ~30 chars; "calemdar: " is 10 of them. Fix: optional cfg knob notifications.title_prefix (default "calemdar: ") so users can shorten it.

    N4. buildActionEnv exposes paths verbatim¶

    internal/notify/scheduler.go:243-253 — CALEMDAR_PATH is the absolute path to the event file. For shell-action users, this needs careful quoting. Document the contract: action authors are responsible for quoting. Fix: add a doc comment.

    N5. Scheduler exports mu only via tick/Run callsites¶

    internal/notify/scheduler.go:63-67 — mu guards lastTick only. The mutex's name is generic; prefer lastTickMu sync.Mutex or use atomic.Pointer[time.Time]. Fix: rename for clarity.

    N6. notify_fired lacks a TTL-by-event-path; full-table prune by fire_at_planned¶

    internal/store/store.go:445-452 — pruning by planned-fire-at means a notif that fired 14 days ago is gone, but a row whose fire_at_planned is in the future (because of a clock-skew mishap) is never pruned. Niche. Fix: bump prune to WHERE fire_at_planned < cutoff OR fired_at < cutoff.

    N7. notify_fired event_path index unused¶

    internal/store/store.go:100 — notify_fired_path is created but the only query uses the full PRIMARY KEY (event_path, notify_index, fire_at_planned), which already covers event_path lookups. Fix: drop the redundant index.

    N8. sql.Open doesn't enable _busy_timeout¶

    internal/store/store.go:35 — under WAL with concurrent writers (CLI delete during daemon tick), database is locked errors can surface. Fix: append &_pragma=busy_timeout(5000) to the connection string.

    N9. migrate swallows non-"duplicate column" ALTER errors but only after the substring match¶

    internal/store/store.go:106-122 — strings.Contains(err.Error(), "duplicate column name") is brittle if the modernc/sqlite driver changes the error text. Fix: query PRAGMA table_info(occurrences) first and only ALTER when the column is missing.

    N10. UpsertSeries ignores JSON marshal errors via _, _ := json.Marshal¶

    internal/store/store.go:128-131 — []string and []int will never fail to marshal, so this is fine. Document the assumption with a // json.Marshal of slices cannot fail comment so future fields don't break the invariant silently.

    N11. Series.scanSeries quietly returns nil for sql.ErrNoRows¶

    internal/store/store.go:465-488 — same shape repeated in GetSeriesByPath and ListSeries. Fix: extract a scanRoot helper accepting Scanner, removing ~80 lines of duplication.

    N12. vault.Resolve returns absolute path but doesn't symlink-resolve¶

    internal/vault/vault.go:24-49 — if vault is a symlink, watcher events arrive with the symlink-target path while EventsDir() returns the symlink path. Mismatch breaks classify. Fix: filepath.EvalSymlinks before storing in Vault.Root.

    N13. expandTilde swallows the os.UserHomeDir error¶

    internal/vault/vault.go:68-80 — if expanded, err := expandTilde(p); err == nil { p = expanded } in Resolve — silent fallthrough means a homedir-less environment misroutes "~" to literal "~". Fix: surface the err.

    N14. runEventList parses dates with time.UTC while reading via model.Location()¶

    cmd/calemdar/cmd_event.go:38 — model.ParseDate(date, time.UTC) accepts user input but the rest of the code uses model.Location(). For YYYY-MM-DD parse this is harmless (no DST or offset effect), but the inconsistency is jarring. Fix: pass model.Location() everywhere user-input dates flow.

    N15. Notifications.Validate warns about ntfy topic but not URL when notifs disabled¶

    internal/config/config.go:218-227 — the disabled path checks topic regex but skips URL syntax. Fix: add a URL syntax check for symmetry, or drop the topic pre-flight to keep "disabled means lax".

    N16. ParseLead accepts bare integers as minutes — undocumented in NotifyEntry comment¶

    internal/model/notify.go:34-38, 92-115 — the notify example shows "5m" but "5" works the same. Fix: add the "5" form to the doc comment.

    N17. parseEntry (backup) trusts that slug doesn't contain a hyphen-stamp lookalike¶

    internal/backup/backup.go:138-161 — slug is enforced lowercase by Slugify, which produces hyphens. A slug ending in numerals followed by -2026-... would parse as if 2026-... were the stamp. Slug regex (none) → defensive parsing. Fix: validate the recovered slug against a regex; reject if it ends in -2026 or similar.

    N18. archive.Run doesn't update sqlite occurrences after move¶

    internal/archive/archive.go:70-78 — file moves to archive but occurrences table still points to the old path. Next tick the watcher emits Deleted on the old path → DeleteOccurrence. So eventually consistent, but for a window the cache is wrong. Fix: explicitly DeleteOccurrence(oldPath) (no UpsertOccurrence(newPath) since archive isn't watched).

    N19. runNightly doesn't UpsertOccurrence after reconcile¶

    internal/serve/nightly.go:24-35 — only UpsertSeries is called; the occurrences from the reconcile aren't pushed into sqlite. The next watcher tick eventually surfaces them, but until then the cache is behind. Fix: mirror dispatchRecurring's full upsert loop.

    N20. actions.curateEnv lets caller-supplied extra overwrite any earlier-set key¶

    internal/actions/actions.go:259-278 — extra map is appended last; an action that sets CALEMDAR_TITLE=foo via env can collide with PATH if a buggy CALEMDAR_* mapping ever overlaps. Today no overlap exists. Fix: explicitly forbid extra keys that match desktopPassthrough or PATH/HOME/USER.

    N21. runEventList has range flag default "week" but no validation of "yesterday"-style misuse¶

    cmd/calemdar/cmd_event.go:114-145 — switch only knows {today, week, month, all}. Returns clear error. OK as-is.

    N22. ParseLead rejects "-5m" correctly but accepts "+5m" as 5m via time.ParseDuration¶

    internal/model/notify.go:107-114 — fine, but undocumented. Fix: document explicitly that leading + is permitted and - rejected.

    Test-coverage gaps (callouts; fold into Important fixes if you ship a test-pass)¶

    • Bulk-fanout reconcile race — no test exercises dispatchRecurring while the watcher's selfWrites map is hot. A test that triggers reconcile of a 100-event series, then immediately os.Stats every written file via the fsnotify pipeline, would catch any regression of the autoown false-flip.
    • Scheduler restart-replay window — TestSchedulerDoesNotReplayHistoryOnRestart covers fire_at strictly in the past. No test for fire_at exactly at now - 2*tickInterval (the boundary).
    • Recurring delete --purge-events cascade — TestRecurringDeleteWithPurgeEvents verifies file removal and backup; no test confirms sqlite DeleteOccurrence actually runs (a follow-up ListOccurrencesInRange to assert empty would close the loop).
    • handleRecurringDelete when sqlite is unreachable — what if r.RawSource == "" AND the file is gone? Logs "snapshot missing"; no test.
    • autoown.FlipIfNeeded on a transiently missing file — model.ParseEvent returns os.ErrNotExist; the dispatch swallows it, but autoown is called before the existence check and may surface its own error. Add a test.
    • Watcher behaviour when fsnotify drops events — kernel buffer overflow path is untested. A queue-flood test would catch any regressions in the Errors channel handling.
    • writer.WriteRoot self-write notify ordering — current code calls writeMarkdown then NotifySelf. No test confirms that the watcher actually sees mtime equality (write vs stat race). Fold into watch_test as a property test.
    • reactor.Run slug collision — collision check exists; no test for it.
    • ListUpcomingWithNotify with malformed JSON — current code continues; add a test asserting the malformed row is skipped without panicking.
